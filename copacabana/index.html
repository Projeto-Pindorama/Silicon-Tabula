Operao Fub Cake: compilando o Copacabana do zero

Esse documento contm instrues para que voc possa, por conta prpria, compilar a distribuio Copacabana Linux, podendo desde obter aprendizado at mesmo contribuir ao projeto diretamente com correes de bugs (falhas) ou portando para alguma outra arquitetura.
O processo bem descrito em vrias etapas, para que voc no se perca no caminho, assim como bem explicado ou seja, voc no vai apenas copiar e colar instrues, mas sim entend-las sem muito esforo.

Untando as formas: formatando e montando os discos

Aps montar a partio root (/) da imagem de disco com o L.E.mount, crie pontos de montagem para o /boot, /usr, /opt e /var, que sero utilizados na compilao, e ento monte essas parties com o comando mount(8) padro.

lemount -D /dev/loop0p5 -t dsk
mount /dev/loop0p1 $COPA/boot
mount /dev/loop0p6 $COPA/usr
mount /dev/loop0p7 $COPA/var
mount /dev/loop0p8 $COPA/opt

Se voc desejar, pode pr isso dentro de um arquivo e executar com o Bourne shell, a fim de matar tempo.

Os ingredientes

Essa seo contm uma lista de dependncias necessrias para montar a toolchain intermediria e o sistema-final, tanto aquelas que devem estar presentes na mquina-hospedeira quanto instrues para baixar o cdigo-fonte necessrio.

Pacotes que devem estar instalados na mquina hospedeira

  • Um shell compatvel com os padres POSIX atuais, preferivelmente acompanhado do Korn Shell (KSH-93) ou do GNU Bourne-Again Shell;
  • Ferramentas padres do UNIX, preferivelmente as uutils, lobase ou as GNU coreutils, que seguem os padres mais atuais e que muitos scripts iro requisitar;
  • Mitzune, para o estgio de chroot;
  • L.E.mount, opcionalmente para montar discos;
  • cmp(1), diff(1), diff3(1), sdiff(1) e o patch(1);
  • GNU m4 e as GNU auto*tools;
  • GNU Make;
  • GNU sed, partindo da verso 4.1.5;
  • New AWK/BWK AWK, ou o GNU AWK;
  • util-linux;
  • Ferramentas de arquivamento: tar(1) e cpio(1);
  • As principais ferramentas de compresso e decompresso: bzip2, gzip e xz.

Baixando o cdigo-fonte

Primeiramente, crie o diretrio /usr/src dentro do disco do Copacabana ($COPA) e ento tenha a certeza de que ele tem as permisses do usurio que voc est usando para montar a toolchain intermediria, no nosso caso o baggio:wheel (ou baggio:baggio, caso seu sistema no tenha o grupo `wheel, ex.: no seja o Copacabana em si).

[baggio@S145 ~]$ mkdir -p $COPA/usr/src

Vamos precisar que o usurio `baggio tenha acesso ao diretrio /usr/src (do disco-alvo do Copacabana, s para dar nfase nisso) pois neste diretrio que vamos baixar todo o cdigo-fonte necessrio para compilar todos os trs estgios. Agora apenas execute o script cmd/download_sources.bash, passando os arquivos sources.txt e sources.sha256 como parmetros.

Primeiramente, clone o repositrio do Copacabana pelo git:

[baggio@S145 ~]$ git clone http://github.com/Projeto-Pindorama/copacabana.git \
    $COPA/usr/src/copacabana

Aps isso, entre no diretrio $COPA:

[baggio@S145 ~]$ cd $COPA/usr/src/copacabana

E ento execute o script:

[baggio@S145 copacabana]$ bash cmd/download_sources.bash sources.txt sources.sha256

Nota: recomendvel que, enquanto voc estiver baixando o cdigo- fonte, rode o comando watch -n1 ls -lR $COPA/usr/src, para ver o que est sendo baixado e se o script est funcionando.

Tambm crie, aps o download, um diretrio chamado cmp/, onde vamos extrair o cdigo-fonte para compilar.

[baggio@S145 copacabana]$ mkdir $COPA/usr/src/cmp

Isso no s por uma questo de portabilidade, pois algumas implementaes do tar(1) no tm suporte descompresso on-the-fly como, por exemplo e at onde eu inicialmente testei, a implementao encontrada no OpenBSD1 mas tambm uma questo de criar o (bom) costume de no jogar tudo nas mos do programa que estiver usando.

No exemplo com o arquivo.tar, vamos supor que ele tenha criado um diretrio chamado diretorio-arquivo-tar; entre nesse diretrio e ento execute as instrues dadas aqui no manual, pois eu estou assumindo que voc esteja com o diretrio de trabalho ($PWD) como o diretrio com o contedo da tarball, no como o /usr/src, /usr/src/cmp ou qualquer outra coisa.

Biblioteca C musl

Pois bem, essa parte no bem explicada tanto pelo Linux from Scratch quanto pelo Musl-LFS, mesmo assim vou tentar dar uma explicao sobre o porqu de compilarmos a musl para essa toolchain e no deixarmos tudo apenas para o final. Precisamos compilar a musl agora pois, alm do fato da /cross-tools sim, nossa amada toolchain inicial com apenas um compilador, biblioteca C e cabealhos no estar disponvel no nosso prefixo da Mitzune, ns estaremos linkeditando bibliotecas dinmicas nos nossos binrios compilados nessa etapa, essas que devem apontar para nossa toolchain intermediria, no para o sistema-base (ex.: /lib/ld-musl-x86-64.so.1) e nem para a toolchain inicial (/cross-tools/lib/ld-musl-x86_64.so.1), caso contrrio teremos o clssico erro de bash: X: No such file or directory, que basicamente quando o interpretador ELF (que, se minha memria no falha, o ldd(1)) no encontra a biblioteca que o binrio pede para ser executado.

Aps essa explicao curta, vamos ao que interessa: compilar a biblioteca C e instal-la no diretrio correto.

1: Rode o script configure

./configure \
    CROSS_COMPILE=${COPA_TARGET}- \
    --prefix=/ \
    --target=${COPA_TARGET}

2: Compile e instale na toolchain

Compile e instale com o GNU make, como um pacote normal.

gmake -j$(grep -c 'processor' /proc/cpuinfo) \
    && DESTDIR=/tools gmake install

Aps instalado, refaa a ligao do /tools/lib/ld-musl*.so.1 para o /tools/lib/libc.so.

rm -vf /tools/lib/ld-musl-$(uname -m).so.1
ln -s /tools/lib/libc.so /tools/lib/ld-musl-$(uname -m).so.1

Isso vai criar uma nova ligao simblica, dessa vez para o libc.so na localizao correta.

Nota para compilaes futuras: Possivelmente a gente poderia evitar isso usando --prefix=/tools, o que faria com que automaticamente os links fossem feitos direto de /tools/lib/ld-musl-*.so.1 para /tools/lib/libc.so e que no precisssemos usar o DESTDIR na hora da instalao. No tenho certeza se funcionaria, todavia algo a se tentar no futuro.

E, por fim, crie um arquivo para que o linkeditor possa se achar e encontrar as nossas bibliotecas da toolchain.

mkdir -pv /tools/etc/ \
&& cat > /tools/etc/ld-musl-$(uname -m).path << "EOF"
/tools/lib
EOF

Ajuste bsico na toolchain inicial antes de continuar

Antes de continuarmos, necessrio que ns ajustemos o GCC da toolchain inicial para usar as bibliotecas da nossa toolchain intermediria.
Esse processo anlogo para no dizer que literalmente o mesmo, s que menos extenso que o apresentado na parte do sistema-base, ento no entrarei em detalhes sobre o que o arquivo specs e afins por hora.

Exporte essas variveis no seu shell:

LIBGCC_PATH=$(dirname $(${COPA_TARGET}-gcc -print-libgcc-file-name))
SPECFILE="$LIBGCC_PATH/specs"

Agora, faa um despejo do specs padro do GCC:

${COPA_TARGET}-gcc -dumpspecs > specs.tmp

Ento faa a alterao do caminho para a biblioteca de /lib para /tools/lib:

sed 's@/lib/ld-musl-x86_64.so.1@/tools/lib/ld-musl-x86_64.so.1@g' < specs.tmp > specs

Nota: O padro de substituio muda de arquitetura para arquitetura. No futuro haver possivelmente uma tabela para cada arquitetura alm de x86 de 64 bits.

Agora, por fim, mova o nosso specs para a nova localizao:

mv ./specs "$SPECFILE"
unset LIBGCC_PATH SPECFILE

Opicionalmente, aps esse passo, voc pode fazer uma prova real para checar se, de fato, a biblioteca C linkeditada a da toolchain.

Primeiro, gera-se um programa simples para ser compilado pelo GCC. Pode ser uma funo main() vazia, como no Musl-LFS do Derrick, ou, como nessa tabula, um clone do comando true(1).

printf 'int main(void) {\n  return 0;\n}\n' > true.c

Em seguida, compila-se o mesmo e faz-se uma leitura de suas propriedades com o ${COPA_TARGET}-readelf.

${COPA_TARGET}-gcc true.c -o true
${COPA_TARGET}-readelf -l ./true | grep 'Requesting program interpreter'

Essa leitura deve retornar o seguinte resultado:

      [Requesting program interpreter: /tools/lib/ld-musl-x86_64.so.1]

Nota: Como dito anteriormente, o nome da biblioteca ir variar para cada arquitetura (ex.: para 64 bits, ser ld-musl-x86-64.so.1; para 32 bits, ser ld-musl-i386.so.1; para o Apple M1 (ou qualquer outro ARM de 64 bits) ser ld-musl-aarch64.so.1 etc).

GNU Binary Utilities (Binutils)

No creio que seja necessrio explicar a importncia das GNU Binutils tanto nessa toolchain quanto no sistema-base. Um exemplo da importncia desse pacote que ele contm o linkeditor.

1: Exporte as variveis de ambiente ao fim do ~/.bashrc

V ao ~/.bashrc (aquele que copiamos do repositrio de desenvolvimento do Copacabana ao comeo da compilao cruzada) e, ao fim do arquivo, descomente (esse verbo existe formalmente? No fao ideia) essa linha:

export CC CXX AR AS RANLIB LD STRIP

E ento comente essa:

unset CC CXX AR AS RANLIB LD STRIP

Feito isso, apenas recarregue o arquivo no Shell, como de costume.

. ~/.bashrc

Isso far com que o GNUMakefile e o configure chamem as nossas ferramentas da toolchain inicial ao invs das da mquina hospedeira.

2: Para targets de 64 bits, um hack com amor

Caso voc esteja compilando de (e para) uma mquina de 64 bits x86, faa um hack no sistema de arquivos da toolchain intermediria entre o diretrio lib/ e o lib64/.

ln -s /tools/lib /tools/lib64

Isso, no sistema-base, feito com binding entre os dois diretrios, mas como no vamos nos dar ao trabalho de chamar o mount() s para algo que , a grosso modo, um grande hack, simplesmente faa isso.
Como diria um veterano que conheci, Dont overthink.

3: Rode o script configure

Por algum motivo, antes de se rodar o script, o projeto GNU recomenda que voc rode o script configure em um diretrio dedicado.
Essa uma peculiaridade curiosa ou, para ser honesto como quem j manteve pacotes (extra-oficiais) no Slackware e agora mantm os pacotes oficiais no Copacabana, uma bela de uma gambiarra no processo de compilao da maioria dos, digamos, carros-chefe do projeto GNU, como o GCC, o GNU ncurses ou as prprias GNU Binutils.

mkdir build \
&& cd build \
&& sh ../configure --prefix=/tools      \
             --with-lib-path=/tools/lib \
             --build=${COPA_HOST}       \
             --host=${COPA_TARGET}      \
             --target=${COPA_TARGET}    \
             --disable-nls              \
             --disable-werror           \
             --with-sysroot

4: Compile e instale na toolchain

gmake -j$(grep -c 'processor' /proc/cpuinfo) && gmake install

O prximo passo instalao do pacote das Binutils que criemos um novo binrio para o linkeditor, chamado ld-new.
Essa parte melhor explicada no ajuste dessa toolchain para a compilao do sistema-base, mas, dando uma palinha, basicamente vamos criar um binrio do linkeditor com caminhos diferentes para procurar bibliotecas.
O nosso linkeditor atual tem o /tools/lib como o caminho para se buscar bibliotecas pedidas numa compilao, o que o correto e necessrio para essa etapa, mas o nosso novo linkeditor ter o /lib e o /usr/lib como caminhos a se buscar.
Sem mais delongas, rode esses comandos:

gmake -C ld clean
gmake -C ld LIB_PATH='/lib:/usr/lib'

Isso vai fazer com que o gmake aja apenas dentro do diretrio ld/ (a opo -C no gmake equivale ao chdir()), presente dentro da rvore de cdigo-fonte das Binutils, limpe os objetos e binrios antigos e ento compile um novo binrio para o linkeditor com um novo LIB_PATH manualmente configurado por ns fun fact: pelo configure, o LIB_PATH originalmente foi exportado globalmente como /tools/lib por meio do switch --with-lib-path.

Ento, por fim, copie o novo binrio do linkeditor para a toolchain:

cp ld/ld-new /tools/bin

GNU Compiler Collection (GCC)

O GNU Compiler Collection um pacote que contm compiladores para diversas linguagens. Inicialmente, surgiu como uma reescrita feita pelo Projeto GNU do compilador C, conhecido como cc(1) no UNIX da AT&T e, por conta disso, ganhou o nome de gcc, pois os comandos (re)escritos pelo GNU usavam o prefixo g. Logo gcc seria a verso GNU do comando cc(1).
Estaremos compilando os compiladores C e C++ do pacote, o que o suficiente, por hora enquanto as futuras ferramentas em Go no so escritas , para compilar todo o sistema-base.

1: Bibliotecas extras (GMP, MPC e MPFR)

O GCC depende nas bibliotecas GMP, MPC e MPFR, que so usadas para que o GCC possa lidar com aritmtica de nmeros complexos, preciso arbitrria e afins.
Elas no so opicionais e devem ser extradas para dentro da rvore de cdigo-fonte do GCC.

Para isso, apenas extraia cada uma delas dentro do diretrio do GCC.
O comando para extrair basicamente o mesmo que voc usou para extrair o GCC em si, mas dessa vez voc deve usar o switch -C passando o diretrio do GCC como parmetro.

Os comandos finais ficariam assim:

xz -cd $COPA/usr/src/pkgs/gnu/gmp-6.2.1.tar.xz | tar -xvf - -C ./gcc-10.3.1_git20210424/
gzip -cd $COPA/usr/src/pkgs/gnu/mpc-1.2.1.tar.gz | tar -xvf - -C ./gcc-10.3.1_git20210424/
xz -cd $COPA/usr/src/pkgs/gnu/mpfr-4.1.0.tar.xz | tar -xvf - -C ./gcc-10.3.1_git20210424/

2: Aplique os patches do Alpine Linux

Por ltimo, mas longe de menos importante, devemos apenas trocar o caminho padro de bibliotecas que o gcc(1) (no o pacote GCC, mas sim o binrio gcc) vai usar na hora de linkeditar nossos programas (de forma paralela ao linkeditor das GNU Binutils, o ld(1)), por meio da definio de um macro chamado STANDARD_STARTFILE_PREFIX_1. O pequeno hack em shell script abaixo faz o servio. Ele foi adaptado do livro original do Linux from Scratch assim como uma parte considervel dos hacks aqui presentes, com algumas pequenas mudanas, como a clarificao da linha de comando do sed e a explicao do que de fato os comandos ali fazem.

for i in gcc/config/{linux,i386/linux{,64}}.h; do
    # Copia o arquivo apenas se o arquivo tiver sua data de
    # criao/edio mais recente que a do arquivo-alvo (ou
    # se o arquivo-alvo no existir, obviamente).
    cp -u "$i" "$i.orig"
    sed -e 's@/lib\(64\)\?\(32\)\?/ld@/tools&@g' \
        -e 's@/usr@/tools@g' "$i.orig" > "$i"
    cat >> "$i" << "EOF"
#undef STANDARD_STARTFILE_PREFIX_1
#undef STANDARD_STARTFILE_PREFIX_2
#define STANDARD_STARTFILE_PREFIX_1 "/tools/lib/"
#define STANDARD_STARTFILE_PREFIX_2 ""
EOF
    # Altera a data de criao/edio do arquivo .orig,
    # assim nos impede de dar um tiro no p se rodarmos
    # o lao novamente --- no caso, sobreescrevermos o
    # arquivo original de facto com o arquivo editado
    # que recm criamos, pois o arquivo original agora
    #  mais recente que o arquivo editado.
    touch "$i.orig"
done

Acima temos um encanamento ligando entrada e sada, onde a entrada o comando no caso, a sada dele gcc com a opo -dumpspecs que, como voc deve estar imaginando, exporta as specs padres do GCC que compilamos anteriormente, passa por uma linha de comando do sed (a qual irei explicar daqui h pouco) e termina com a sada indo para o um specs.tmp. Se est se perguntando Por que no salvamos direto como o $SPECFILE direto?, isso ser respondido no passo seguinte.
Essa linha de comando do sed faz o seguinte:

  • s@/tools@@g: Remove todas as menes ao /tools, no caso trocando por nada usando o comando de substituio;
  • /\*startfile_prefix_spec:/{n;s@.*@/usr/lib/ @}: Busca pelo padro *startfile_prefix_spec:, ento substitui tudo (.*) abaixo dele (n seria a quebra de linha, anlogo ao \n) por outro padro (/usr/lib) uma vez que o tal padro dito encontrado (note que no h o g ao fim, que indica que a mudana no deve ser global);
  • /\*cpp:/{n;s@$@ -isystem /usr/include@}: Faz, assim como a linha acima, uma busca por padro (este sendo o *cpp: no caso) e ento substitui o fim da linha ($) abaixo desse padro (de novo h a quebra de linha usando n) por um novo padro (-isystem /usr/include) uma nica vez tambm (note, novamente, que no h a indicao de que deva ser uma mudana global).

Meu ld na segunda parte da compilao da biblioteca C musl no sistema-base no est retornando o resultado esperado

Enquanto eu compilava o sistema-base do Copacabana no dia 22 de maro de 2022, com intuito de j fazer o primeiro lanamento oficial, eu dei de cara com esse problema:

SEARCH_DIR("=/tools/x86_64-pindoramaCOPACABANA-linux-musl/lib64")
SEARCH_DIR("/tools/lib")
SEARCH_DIR("=/tools/x86_64-pindoramaCOPACABANA-linux-musl/lib")

Se voc j leu sobre essa parte da compilao, possivelmente percebeu o erro tratado aqui: o linkeditor no est usando os caminhos corretos para as bibliotecas, mas sim os caminhos antigos da toolchain.
Antes de continuar, se pergunte se voc fez o reajuste e substituiu o binrio do linkeditor na parte de reajuste, se no, faa que o erro deve sumir.
Caso contrrio, venha comigo para a aventura de recompilar (parte d)a toolchain pois possivelmente voc possivelmente esqueceu de limpar o diretrio do ld dentro do diretrio build/ das GNU Binutils antes de criar o ld-new, logo acabou com dois binrios com nomes diferentes, mas com uma mesma configurao.
Para corrigir isso, apenas refaa a parte do novo binrio para o linkeditor nas instrues para compilar as GNU Binutils para a toolchain, s no se esquea de limpar o diretrio antes com o gmake -C ld/ clean.

Esse erro originalmente foi percebido pelo prprio Derrick no Musl-LFS, enquanto ele trabalhava na verso 10.0.0-RC1, e corrigido no commit 19e881cd880ecd6fc8a6711c1c9038c2f3221381 no dia 12 de dezembro de 2021.[12]

O grande problema est no ponto em que sim, a biblioteca C musl tem sim uma implementao da setns(), que acaba conflitando com a implementao do iproute2 e dando esse erro na compilao, como se a funo setns(int, int) em si tivesse sido declarada duas vezes.
Alm de ter sido experenciado por mim mesmo e, provavelmente qualquer um que usou a toolchain de 25 de janeiro de 2022 , tambm documentado num commit feito em 16 de Julho de 2011 por Eric W. Biederman na rvore de desenvolvimento principal do iproute2, implementando originalmente a funo check_setns() no script configure, a fim de resolver um outro erro extremamente similar ao nosso, mas que foi provocado por outros fatores que havia sido experienciado por Dan McGee, um desenvolvedor de software de Chicago que trabalhou no projeto Arch Linux como desenvolvedor do gerenciador de pacotes Pacman e mantenedor do website do projeto[], enquanto ele empacotava o iproute2[].

Infelizmente, por uma questo de tempo e pacincia, eu acabei por no conseguir traar a causa real desse erro.
Entretanto, eu pude criar algumas teorias, e citarei duas delas aqui, da mais provvel para a menos provvel:

  • M-configurao do arquivo specs do GCC no reajuste, somado ao linkeditor quebrado o que poderia ser questionado pois, mesmo assim, o pr-processador ainda sim encontra o cabealho sched.h contendo a implementao do setns(int, int), ento de alguma forma os caminhos para o diretrio de cabealhos esto configurados corretamente, todavia essa hiptese ainda a mais plausvel;
  • Falha na execuo do script em si, por falta de ferramentas presentes na toolchain ou por essas no seguirem padres atuais do POSIX.

De qualquer forma, eu consegui corrigir esse erro utilizando a toolchain mais recente, de 1 de Julho de 2022; agora o script de configurao do iproute2 (e de outros programas) finalmente consegue identificar a presena de funes na biblioteca C.

Soluo secundria

Caso voc esteja usando a toolchain antiga, algo que no recomendo pois voc pode acabar quebrando outros programas, a soluo para esse erro seria (re)forar a presena da setns(), editando o script configure e removendo (ou simplesmente comentando) a parte da funo check_setns() que, de fato, faz a checagem, como pode ser visto abaixo:

check_setns()
{
        echo "IP_CONFIG_SETNS:=y" >>$CONFIG
